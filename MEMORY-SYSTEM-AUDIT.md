## Part 2: Auto-Memory — How It Works

### What it is

A file-based system where Claude writes notes about the current project to persist across sessions. Think of it as Claude's notebook for things you'd otherwise have to re-explain every session.

### Memory types

| Type | What it captures | Filename pattern |
|------|-----------------|-----------------|
| `user` | Your role, expertise level, preferences, working style | `user_*.md` |
| `feedback` | Corrections, what to avoid, what worked — behavioral guidance | `feedback_*.md` |
| `project` | Ongoing work context, decisions, active bugs, deadlines | `project_*.md` |
| `reference` | Pointers to external systems (Linear boards, Grafana dashboards, Slack channels) | `reference_*.md` |

### What triggers a write

**No automatic trigger.** Claude writes memories when:

1. You explicitly ask: "remember this", "save that", "note that for future sessions"
2. Claude judges something worth preserving during conversation — learning your role, hearing a correction, understanding why a decision was made

There is no scheduled save, no end-of-session trigger, no threshold. It fires on Claude's judgment or your instruction.

### What's in context each session

When a session starts, the harness automatically injects `MEMORY.md` as a system reminder. This is a lightweight index — one line per memory — not the full file contents. Example of what currently loads:

```
- [Dotfiles and marketplace layering](project_repo_layering.md) — Three-layer Claude Code config
- [MatterMore agents repo](reference_agents_repo.md) — mattermoreai/agents is the team's marketplace
- [R style enforcement](project_r_enforcement.md) — Four-layer R linting...
- [CodeRabbit triage pipeline](project_coderabbit_triage.md) — Shell wrapper + hook + /coderabbit-fix
- [Session handoff 2026-04-27](handoff_2026-04-27_20fc8532.md) — manual compact, branch main
```

Claude reads full memory files on demand — when a topic is relevant, it reads the file for detail. The index tells Claude what exists; Claude decides what to fetch.

### Context budget

- The MEMORY.md index is always loaded. Lines after 200 are silently truncated.
- Individual memory files are loaded by Claude when relevant — not automatic.
- Current state: 6 index lines. No truncation risk.

### Cleanup

**Entirely manual.** There is no TTL, no auto-archival, no cleanup rule. Stale memories accumulate unless you explicitly ask Claude to delete or update them. The harness will remind Claude that memories over a certain age may be outdated.

### What's currently missing

The dotfiles project memory has no `user_` or `feedback_` files. That means Claude has no persisted understanding of:
- Who you are (I-O Psych researcher, multi-machine, multi-project)
- Behavioral corrections (e.g., terse output, no em dashes, stop hook = "Ok")
- Preferences that should carry forward

These are being captured inline each session from CLAUDE.md rules, but they aren't stored as memories. The distinction matters: rules define constraints, memories define context. "Don't use em dashes" is a rule. "This user is a senior researcher comfortable with R/Python/SQL who doesn't need basic explanations" is a memory.

## Part 3: Handoffs — How They Work

### What they are

Handoffs are session-state snapshots written immediately before context compaction. Unlike memories (which Claude writes based on judgment), handoffs are machine-generated by shell hooks — they capture objective state, not interpreted context.

### Trigger

The PreCompact hook fires when:
- You manually run `/compact`
- Auto-compact threshold is hit (currently disabled: `autoCompactEnabled: false`)

**This is the key friction point: handoffs only exist if you compact. If you end a session without compacting, no handoff is written.**

### What a handoff captures (`pre-compact.sh`)

```
- Which files were Read / Written / Edited this session
- Git branch and working tree status (staged, unstaged, untracked)
- Last 5 commits
- Git diff stats (files changed, insertions, deletions)
- Trigger type (manual vs auto), session ID, timestamp
```

It extracts file activity by parsing the session transcript JSONL file (using the `transcript_path` provided in the hook input). This is objective — it's what actually happened, not Claude's interpretation of what happened.

### What happens after compaction (`post-compact.sh`)

The PostCompact hook fires immediately after compaction completes. It reads the most recent handoff file and outputs it as a `systemMessage`. This gets injected into Claude's context as the first thing Claude sees in the new compacted session — before any user message.

This is why after `/compact` you saw the full git state and file list restored in the output. That wasn't Claude remembering — it was the hook injecting that context.

### Handoff retention in MEMORY.md

The pre-compact script keeps the last 4 handoff entries in the index, plus the one it just wrote. The 5th-oldest drops off. Handoff files themselves stay on disk indefinitely — they just stop being indexed.

Current state: 1 handoff file, 1 index entry.

### Scope

Like memories, handoffs are project-scoped. A handoff for the dotfiles project lives in the dotfiles memory directory and only appears in dotfiles sessions.

## Part 4: How They Interconnect

```
Session start
     │
     ▼
MEMORY.md index injected as system-reminder ──► Claude loads relevant
     │                                           memory files on demand
     │
     ▼
Session work (Claude writes memories as needed)
     │
     ├── [optional] /compact triggered
     │        │
     │        ▼
     │   pre-compact.sh fires
     │        │ writes handoff_<date>_<id>.md
     │        │ adds entry to MEMORY.md
     │        ▼
     │   Compaction runs
     │        │
     │        ▼
     │   post-compact.sh fires
     │        │ reads latest handoff file
     │        │ injects as systemMessage
     │        ▼
     │   New compacted context begins with handoff injected
     │
     └── [or] Session ends (Stop hook fires)
              │ checks git status only
              │ no handoff written
              ▼
              Next session starts cold (only MEMORY.md index)
```

The two systems share a single directory and a single index file. The index is the connection point. Both memories and handoffs appear in MEMORY.md with one-line entries; Claude treats them the same way when scanning for relevant context.

## Part 5: Current Gaps

### Gap 1: No end-of-session handoff

If you end a session without `/compact`, nothing is written. The next session starts with only the MEMORY.md index — no record of what files were touched, what was decided, or what the git state was. This is the most significant friction point.

### Gap 2: No global memory

Memories are project-scoped. User preferences (terse output, no em dashes, I-O Psych researcher context) have to be re-established from CLAUDE.md rules every session. Rules are behavioral constraints; user-profile memories would let Claude calibrate explanations, assumptions, and communication style without you having to document everything as a rule.

There is no mechanism for a memory to say "this applies everywhere."

### Gap 3: No `user_` or `feedback_` memories written yet

The five existing memories are all project context (how things were built, how they work). None capture the user profile or behavioral feedback. This means Claude relies entirely on rules for behavioral guidance and has no cross-session memory of corrections.

### Gap 4: No skill-improvement loop

There's signal accumulating every session — corrections, workarounds, patterns — but no mechanism to translate that signal into skill updates. The council identified this as the core unfulfilled goal. `task-observer` attempted to solve this but was a poor fit. The gap remains open.

### Gap 5: Memory cleanup is manual and has no prompts

Nothing reminds you that memories exist or that some may be stale. The 20-day-old `project_repo_layering.md` memory is probably still accurate, but there's no regular review cadence.

### Gap 6: Cross-machine drift

`~/.claude/projects/` is not in dotfiles. Memories don't sync across machines. Handoffs don't sync. Each machine starts fresh except for what's in the tracked dotfiles themselves (rules, skills). If you do significant work on machine A and then switch to machine B, B has no context from A's sessions.

## Part 6: Optimization Plan

Prioritized by friction reduction and value return, from easiest to most involved.

### Priority 1: Add a Stop-hook end-of-session handoff (removes Gap 1)

The Stop hook already fires when Claude ends a session. Right now it only checks git status. Add a companion script that writes a lightweight end-of-session summary to the memory directory.

It doesn't need to be as rich as the pre-compact handoff (no transcript parsing needed). It just needs to capture:
- Git state at session end
- Date and session ID
- Any memory files written this session (available from the session transcript if needed, or can be skipped for simplicity)

This makes every session leave a trace, not just sessions where you `/compact`.

**Implementation:** Add a second command to the Stop hook array in `settings.json`, pointing to a new `stop-session-summary.sh` script.

### Priority 2: Write foundational user and feedback memories (removes Gap 3)

You have a well-defined working style and profile. Write these once and they'll load every session:

- `user_profile.md` — I-O Psychology researcher, multi-machine, multi-project, primary languages R/Python/SQL/GCP
- `feedback_output_style.md` — terse output, no trailing summaries, no em dashes, stop hook = "Ok", no emojis
- `feedback_git_discipline.md` — stage by name, logical groups, imperative mood, never force-push

This is a one-time task, maybe 20 minutes. The payoff is that Claude calibrates correctly from session start without relying solely on rule files.

### Priority 3: Create `rules/principles.md` (council recommendation, removes Gap 4 partially)

The council's first concrete recommendation: a cross-cutting principles file that captures architecture-level insights that should constrain all skill creation and revision. This is distinct from per-skill constraints (which live in the skill files) and behavioral rules (which live in rules/).

Content to start with, drawn from recent sessions:
- Skills reference rule files; never duplicate rule content in a skill
- Skills have trigger fields for explicit-only invocations (`trigger: /skill-name`)
- Descriptions must signal invocation model ("Explicit /skill-name —" or "Auto-triggers when...")
- Rules take precedence over skill instructions when they conflict
- Agent spawn threshold: tasks under 4 tool calls run inline; never spawn for file listing or targeted reads

Maintain it manually after sessions where something generalizable emerges. Review it before writing any new skill or revising an existing one.

### Priority 4: Build a `/skill-review` skill (council recommendation, removes Gap 4 fully)

The council's second recommendation: a demand-triggered skill (not auto-running every session) that:

1. Reads the last N handoff files in the memory directory
2. Reads the feedback memory files
3. Reads the current rules/principles.md
4. Identifies patterns: repeated corrections, recurring workarounds, gaps between rule intent and observed behavior
5. Outputs a concrete proposal: which skill or rule file to change, what to add or remove, and why

The user reviews and commits. This is the skill-improvement loop the council said was the core unfulfilled goal — and it uses existing infrastructure (handoff files, memory files) rather than requiring a new observation log.

Scope: ~150-200 lines. Demand-triggered only (no auto-activation). Output is a proposal document, not automatic edits.

### Priority 5: Address cross-machine drift (removes Gap 6, optional)

The cleanest solution: move the memory directory into dotfiles. This means:
- `~/.dotfiles/claude/memory/` replaces `~/.claude/projects/-Users-dkgreen-phd--dotfiles/memory/`
- Symlink the dotfiles memory path to the expected project memory path
- Changes commit to git and sync to other machines normally

The complication: other projects (not dotfiles) would still have project-specific memories in `~/.claude/projects/`. Those can't be in dotfiles without a more complex symlink scheme. For the dotfiles project (the global config project), moving to dotfiles is clean. For per-project memories, the sync problem remains unless you're OK with them being machine-local.

This is optional — decide based on how much cross-machine consistency matters to you in practice.
