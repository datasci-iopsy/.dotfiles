C — The Contrarian · blast-radius framing, named the load-bearing risk

Security > Blast-radius containment (not generic CIA). Performance > Per-prompt hook tax. Skill correctness > Built-in collision surface. Availability is noise here.

Failure modes: (1) skipDangerousModePermissionPrompt: true AND claude-yolo alias = two independent bypass paths around the PreToolUse hooks blocking *.env*, *credentials*, bq rm, gcloud delete. The blocklist is theater. (2) Three UserPromptSubmit + three PreToolUse hooks per turn = invisible token tax. (3) test-compact-hooks.sh in production hooks. Single biggest risk: the dual permission bypass. Hook-based blocklist is the only thing standing between a hallucinated command and the work GCP project, and there are two routes around it. Five out of five peer reviewers ranked this response strongest specifically for naming this.

D — The First Principles Thinker · this isn't dotfiles, it's a policy stack

This isn't a dotfiles repo. It's a personal Claude Code policy stack with a thin shell-config tail. Eighty percent of the surface (rules, skills, hooks, agents, commands) shapes one LLM's behavior. Once named correctly, Linux portability matters less than whether the stack is internally coherent and the LLM can execute it under load.

"Two profiles on one machine" is wrong — there's one profile plus a machine-local override file. Either build real profile routing or drop the claim.

CIA triad is the wrong frame — single-user dev tooling, not adversarial systems. The real failure modes are instruction collisions, token waste, and skill redundancy.

Proposed rubric: Policy coherence · Execution cost · Skill justification · Boundary hygiene · Portability surface (bash-only).

Wrong question: "is this secure and performant?" Right question: is the stack causing or fixing the friction the user's own insights data already flagged (Claude guessing, jumping to edits, missing conventions)?
